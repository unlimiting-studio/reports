1. 한 줄 thesis

sena-ai v3 는 codex CLI 와 Claude Code 를 핵심 LLM 엔진으로 쓰는 단일 프로세스 에이전트 하네스이며, ai-sdk + chat-sdk 위에 운영 보조 인프라(drain wrapper · steering 레이어 · schedules · channel context middleware)만 얹은 얇은 앱 레이어 다.

모델/대화 라우팅은 외부 라이브러리에 위임하고, 우리 코드(@sena-ai/app)는 그 둘을 엮을 때 비는 운영 안전망과 외부 빌더용 진입 단순화만 책임진다. 한 명의 외부 빌더가 cwd 안에서 sena.config.ts 한 파일 + run() 한 호출로 봇 한 대를 띄울 수 있는 형태다.

2. 외부 의존 스택

packages/app/package.json 의 dependencies 7개를 세 layer 로 묶고, 각 패키지가 우리 코드 어디에서 어떻게 쓰이는지 한 줄씩 정리.

모델 layer LLM 엔진 + 추상화

ai · @ai-sdk/provider · ai-sdk-provider-claude-code · ai-sdk-provider-codex-cli

  • ai (Vercel AI SDK 6.x) — streamText, wrapLanguageModel, stepCountIs, ToolSet 사용. 우리는 핸들러 4 곳 (queue/steering/step-steering/scheduleFanOut) 에서 직접 streamText 를 호출하고, 결과 14 개 background PromiseLike 필드에 silenceStreamTextRejections 로 noop catch 를 박는다.
  • @ai-sdk/provider — LanguageModelV3 타입 / V3 middleware 인터페이스 (specificationVersion: "v3"). channelContext · traceLogger 두 미들웨어가 V3 명세를 구현.
  • ai-sdk-provider-claude-code — starter 의 기본 엔진. claudeCode("sonnet") 한 줄로 Claude Code CLI 위에서 동작 (별도 API 키 없이 cc 로그인 세션 재사용).
  • ai-sdk-provider-codex-cli — codex 엔진을 같은 ai-sdk LanguageModelV3 인터페이스로 노출. config.model 한 줄 교체로 엔진 변경.
채팅 layer transport + state

chat · @chat-adapter/slack · @chat-adapter/state-pg · @chat-adapter/state-memory

  • chat (chat-sdk 4.28.1) — Chat 인스턴스 / onNewMention / onSubscribedMessage / thread.post / channel.post / concurrency: { strategy } / Chat.shutdown. runtime/run.ts 가 이 SDK 의 lifecycle 을 통째로 감싼다.
  • @chat-adapter/slack — Socket Mode 어댑터. 우리 adapters/slack.ts 에서 createSlackAdapter({ mode: "socket", ... }) default 로 한 번 더 wrap 해서 slackAdapter() 로 노출.
  • @chat-adapter/state-pg — Postgres state adapter. starter 가 postgresState({ connectionString }) 로 받고, 부팅 시 5 테이블 자동 생성 (subscriptions / messages / threads 등).
  • @chat-adapter/state-memory — 단위 테스트용. createMemoryState() 로 노출.
운영 layer cron + signals

node-cron

  • node-cron 4.2.1 — setupScheduleFanOut 가 nodeCron.schedule(spec, cb, { timezone: "Asia/Seoul", noOverlap: true }) 로 등록. validate → register → rollback-on-throw 패턴.
  • SIGTERM/SIGINT signal handler 는 Node 기본 process.on(...). 외부 의존 없이 run.ts 가 직접 박는다 (autoRegisterSignalHandlers 옵션으로 끌 수도 있음).

✱ 이 7 개가 sena-ai v3 의 외부 의존 전부다. 우리 코드는 이 위에 얇은 wrapper / middleware / signal-orchestration 만 얹는다.

3. @sena-ai/app 패키지 — 디렉토리와 5 역할

packages/app/src/ 트리. 각 파일이 무엇을 책임지는지 한 줄씩. 

src/
├── config.ts                    ← defineConfig() — SenaConfig 타입 + 기본값 채움
├── env.ts                       ← requiredEnv() — 누락 시 fail-fast
├── adapters/
│   └── slack.ts                 ← slackAdapter() — @chat-adapter/slack thin wrapper (socket default)
├── state/
│   ├── index.ts                 ← postgresState / createMemoryState 재export
│   ├── postgres.ts              ← createPostgresState — connectionString 단순화
│   └── memory.ts                ← createMemoryState — 테스트용
├── middlewares/
│   ├── index.ts                 ← channelContext + traceLogger 재export
│   ├── channel-context.ts       ← 채널별 memory.md / channels.json 자동 system prompt 주입
│   └── trace.ts                 ← turn-start / chunk 분포 stdout trace
├── schedules/
│   ├── index.ts                 ← cronSchedule 재export
│   └── cron.ts                  ← cronSchedule({ name, cron, target, prompt }) factory
└── runtime/
    ├── run.ts                   ← run() — 모든 하부 인프라 통합 entry
    ├── drain.ts                 ← createDrainController — chat-sdk shutdown drain 부재 보완
    ├── steering.ts              ← SteeringRegistry — thread 별 AbortController 슬롯
    ├── stream.ts                ← safePostStream — chat-sdk 부수 발견 #1 우회
    ├── abort.ts                 ← isAbortError / isChatStreamCloseNoise / abortableSleep
    ├── turn-context.ts          ← AsyncLocalStorage 기반 channelId/threadId/trigger propagation
    ├── state.ts                 ← StateInput → chat-sdk StateAdapter 변환
    ├── scheduleFanOut.ts        ← node-cron 등록 + KST + drain.track + noOverlap + rollback
    └── handlers/
        ├── queue.ts             ← chat-sdk concurrency=queue (인터럽트 없음)
        ├── steering.ts          ← 즉시 abort + 새 컨텍스트 재시작 (기본)
        ├── step.ts              ← tool-result chunk 경계에서 abort (mid-tool-call 회피)
        ├── types.ts             ← ChatSdkHandler / HandlerDeps / gracefulDrainSkip
        └── utils.ts             ← stripLeadingMention · buildPromptWithSkipped ·
                                    silenceStreamTextRejections · tapTextDelta

5 역할 요약

  1. (1) 통합 entry — run() 한 호출이면 chat-sdk Chat 인스턴스 + ai-sdk middleware chain (wrapLanguageModel) + drain wrapper + steering 레이어 + signal handler + cron schedule fan-out 까지 자동 등록.
  2. (2) 핸들러 3 종 — queue (concurrency=queue + skipped 흡수) / steering (즉시 abort + 새 컨텍스트 재시작 · 기본) / step-steering (tool-result chunk = step 경계에서 abort).
  3. (3) drain + steering 레이어 — chat-sdk 4.28.1 부수 발견 3 건 (drain 부재 · stream stop 시 not_authed · external thread reference stream 깨짐) 을 우리 wrapper 로 흡수.
  4. (4) schedules fan-out — node-cron 기반 cron 트리거. KST 시간대 / lazy file read prompt / drain.track / noOverlap=true / register-rollback. Slack target 분기 (thread vs channel).
  5. (5) factory wrapper — slackAdapter() · postgresState() · createMemoryState() · requiredEnv() 로 외부 빌더가 @chat-adapter/* 직접 import 안 해도 되게 진입 단순화.

4. starter 패턴 — templates/slack-agent/

step 4.6 (cbc0208) 의 핵심 산출물. 외부 빌더가 이 디렉토리를 복제하고 env 3개만 채우면 lily 같은 봇이 즉시 도는 형태. templates/slack-agent/src/index.ts 전체: 

import { defineConfig, requiredEnv, run } from "@sena-ai/app";
import { slackAdapter } from "@sena-ai/app/adapters/slack";
import { channelContext, traceLogger } from "@sena-ai/app/middlewares";
import { cronSchedule } from "@sena-ai/app/schedules";
import { postgresState } from "@sena-ai/app/state";
import { claudeCode } from "ai-sdk-provider-claude-code";

const config = defineConfig({
  cwd: import.meta.dirname,
  model: claudeCode("sonnet"),
  adapters: [
    slackAdapter({
      appToken: requiredEnv("SLACK_APP_TOKEN"),
      botToken: requiredEnv("SLACK_BOT_TOKEN"),
    }),
  ],
  middlewares: [
    channelContext({
      cwd: import.meta.dirname,
      channelsFile: ".sena/channels.json",
      memoryDir: ".sena/channels",
    }),
    traceLogger({ label: "sena" }),
  ],
  schedules: [
    cronSchedule({
      name: "morning-briefing",
      cron: "0 8 * * *",
      target: { type: "slack-channel", id: "C0YOURCHANNEL" },
      prompt: { file: ".sena/prompts/morning-briefing.md" },
    }),
  ],
  state: postgresState({ connectionString: requiredEnv("DATABASE_URL") }),
});

const app = await run(config, { steerMode: "steering" });

for (const signal of ["SIGTERM", "SIGINT"] as const) {
  process.once(signal, () => {
    void app.shutdown().then(() => process.exit(0));
  });
}

요약:

  • env 3 개: SLACK_APP_TOKEN · SLACK_BOT_TOKEN · DATABASE_URL. claudeCode("sonnet") 는 cc CLI 로그인 세션을 재사용하므로 별도 키 불필요.
  • 외부 빌더가 SDK 객체를 직접 다루지 않는다 — @chat-adapter/* / chat 패키지를 직접 import 하지 않고 우리 factory wrapper 로만 받는다.
  • app.shutdown() 은 chat.shutdown + drain + signal handler 정리까지. process.exit 만 호출자가 명시 (라이브러리 임베드 시나리오 보호).

5. 운영 흐름 — mention 한 번 → 응답까지

Slack 사용자 멘션이 들어와 응답이 나갈 때까지의 데이터 흐름.

① Slack 사용자 멘션
        │ (Socket Mode)
        ▼
② @chat-adapter/slack  ──→  chat-sdk Chat
        │                        │
        │   chat-sdk dispatcher  │
        │                        ▼
        │              concurrency=concurrent (steering 위임)
        │                        │
        ▼                        ▼
③ chat.onNewMention(thread, message, ctx)
        │
        │  ┌───────────────────────────────────────────────┐
        │  │ run.ts 가 등록한 wrapper:                      │
        │  │  • drain.track("onNewMention")  ─┐            │
        │  │  • thread.subscribe()            │ inFlight++ │
        │  │  • runWithTurnContext({          │            │
        │  │      adapter:"slack",            │            │
        │  │      channelId, threadId,        │            │
        │  │      trigger:"mention" })        │            │
        │  └───────────────────────────────────────────────┘
        │
        ▼
④ steering handler (steerMode 기본):
        │
        │  • gracefulDrainSkip (drain 중이면 안내 후 빠짐)
        │  • SteeringRegistry.set(threadKey, { controller, ... })
        │  • prev 슬롯 있으면 prev.controller.abort() + 안내 post
        │  • streamText({ model: wrappedModel, prompt, tools, stopWhen })
        │  • silenceStreamTextRejections(result)  ← background 14 필드 noop catch
        │
        ▼
⑤ ai-sdk wrapLanguageModel chain (transformParams):
        ├─ channelContext.transformParams
        │     └─ getTurnContext().channelId 기준 channels.json + memory.md 로드
        │        → system message append
        └─ traceLogger.transformParams / wrapStream
              └─ "[sena] turn.start type=... chars=..."
        │
        ▼
⑥ LanguageModelV3 (claudeCode / codex-cli)
        │
        ▼   fullStream (text-delta · tool-call · tool-result · finish ...)
        │
⑦ safePostStream(thread, result):
        ├─ thread._currentMessage.author.userId 있으면 → thread.post(fullStream)  (streaming)
        └─ 없으면 (외부 reference)             → await result.text → thread.post(string)
        │
        ▼
⑧ chat-sdk → state-pg 에 응답 메시지 기록 + Slack API 송신
        │
        ▼
⑨ drain.track finally: inFlight--
   SteeringRegistry.releaseIf(threadKey, slot)

핵심 포인트:

  • turn-context는 AsyncLocalStorage 라 핸들러가 호출하는 모든 ai-sdk middleware / tool 이 같은 turn 의 channelId/threadId 를 read-only 로 본다.
  • 두 번째 멘션이 같은 thread 에 도착하면 ④ 단계에서 prev slot 의 controller.abort() → ⑥ 의 streamText 가 AbortError 로 마감 → 새 slot 으로 재진입. 이때 reject 되는 background PromiseLike 14 개를 ⑦ 직전의 silenceStreamTextRejections 가 미리 noop catch 등록해 둬서 process kill 회피 (5/10 회귀 fix).
  • SIGTERM 이 들어오면 drain wrapper 가 inFlight=0 까지 기다린 뒤 chat.shutdown() 호출. process.exit 은 호출자 책임.

6. schedules fan-out 흐름

cron 트리거는 incoming message 가 없는 stand-alone trigger 라서 mention 흐름과 분기가 다르다. 

① run() 안에서 setupScheduleFanOut(schedules, deps)
        │
        │  Phase 1: 모든 spec validate (cron 표현 + target 형태)
        │  Phase 2: nodeCron.schedule(spec, cb, {
        │            timezone: "Asia/Seoul",
        │            noOverlap: true,
        │          })
        │  실패 시 already-registered task 들 stop+destroy 로 rollback
        │
        ▼
② cron tick (예: "0 8 * * *" KST)
        │
        ▼
③ if (stopping || drain.draining) → skip (shutdown 진입 시 race-safe)
        │
        ▼
④ drain.track(`schedule:${name}`, async () => { ... })
        │
        │  • promptText = lazy fs.readFile (재시작 없이 prompt 변경 반영)
        │  • runWithTurnContext({
        │      adapter:"slack",
        │      channelId, threadId(if thread target),
        │      trigger:"schedule"
        │    })
        │  • streamText({ model: wrappedModel, tools, stopWhen, prompt })
        │  • silenceStreamTextRejections(result)
        │  • text = await result.text         (streaming 미리보기 없음 — 외부 reference 라 #1 회피)
        │
        ▼
⑤ target 분기:
        ├─ kind:"thread"  → chat.thread(`slack:${id}:${threadTs}`).post(text)
        └─ kind:"channel" → chat.channel(`slack:${id}`).post(text)
        │
        ▼
⑥ chat-sdk 가 state-pg 에 일반 mention 응답과 똑같이 기록

step 4.5 history-aware 롤백 (한 줄 요약): cron 의 본질은 incoming message 없는 trigger. thread/channel history 를 prompt 에 합치면 ① 채널 dispatch 시 임의 잡담이 모델 input 에 섞이고 ② thread dispatch 시 prompt 가 thread 에 안 남아 history 가 assistant-only 비대칭 — 두 결함 모두 cron 본질에서 옴. outbound (cron 응답이 state 에 정상 기록)만 충족하고 inbound history-aware 는 step 5+ 결정 보류.

8. 알려진 제약 + step 5+ 영역

  • ai-sdk 내부 unhandled rejection root fix — 우리는 silenceStreamTextRejections 로 known 14 필드만 cover. ai-sdk 가 abortSignal listener 안에서 만드는 내부 promise 까지는 못 잡음. ai-sdk/chat-sdk 내부 PR 영역 → step 5+.
  • Slack triggers.filter(event) 정식 구현 — 4/4 spec 살아있는데 코드 미구현. 작업 thread subscription leak 시나리오의 정공법 → step 5+.
  • conversation ScheduleTarget type — chat-sdk Conversation id 가 transport-scoped thread key 와 같은 형태인지 SPEC 미지수. 1차 fail-fast 유지 → step 5+.
  • @sena-ai/app npm publish — 차니 sign-off 필요. 현재 starter 는 file: 의존으로 동작.
  • cron history-aware — 차니 "incoming 합성" 가설 (visible UX 결정 필요) 결정 보류 → step 5+.
  • mcpServers 통합 — SenaConfig.mcpServers 자리는 박혀있고 fail-fast. provider 옵션 병합으로 실제 연결 → step 5+.
  • attachment-aware 처리 — 텍스트 없는 첨부 메시지는 현재 안내 응답으로 종료. 첨부 다운로드/도구 연결은 step 4+.
