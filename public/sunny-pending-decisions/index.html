결정 1

한국어 종결 판단 (Endpointing)

"사용자 발화가 끝났다"고 판단해 turn.commit을 쏘는 시점 결정. 지금은 마지막 STT final 후 1초 무음만 보고 끝남.

지금 시그널 흐름

STT final("내일 일정 좀 봐줘") → 1000ms 타이머 시작 └─ 1000ms 안에 새 STT final 도착? ─ No → turn.commit 발사 └─ Yes → 텍스트 합치고 타이머 재시작

📍 문제: "내일 일정 좀 봐줘…" 후 사용자가 1초 넘게 생각하다가 "근데 점심은 빼고"를 덧붙이면 → 첫 문장이 이미 commit돼서 응답이 시작됨. "근데"는 별개의 새 turn처럼 처리됨.

결정 후 시그널 흐름

STT final 도착 → 마지막 어절 분류: ├─ continuation marker ("근데", "아니", "그게", "근데", "음") → 타이머 +연장 ├─ commit marker ("해줘", "알려줘", "보여줘") → 타이머 단축 (500ms) └─ 그 외 → 기본 1000ms 유지

📍 마커 목록은 스펙에 박혔어요(continuation 12개 / commit 10개). 연장·단축 ms만 결정하면 됨.

차니가 골라야 할 것 — continuation 마커가 들렸을 때 타이머를 얼마나 늘릴까?

짧은 호흡 한 번 분량.

B안 · +1200ms
추천

총 2200ms 대기.
사람이 다음 어절을 시작하기에 충분.

총 2500ms 대기.
거의 안 끊김. 진짜 끝났을 때 응답 살짝 늦음.

📦 작업량 ~3일 (local-turn-detector.ts)

결정 2

Soft / Hard Interrupt

써니가 말하는 도중(AGENT_SPEAKING) 사용자 끼어들기 처리. 지금은 STT final이 의미 있게 도착해야만 abort.

지금 시그널 흐름

VAD speech_start ──────────────────────────── 무시 (TTS 계속 재생) STT non_final ───────────────────────────── 무시 STT final ("아니 그게 아니라") ──────── 의미 있음(2글자+)? → hard abort

📍 문제: speech_start가 잡혀도 STT final이 도착할 때까지 500~1000ms 동안 써니가 그대로 떠듦. 사용자는 "얘 안 멈추네?" 체감.

결정 후 시그널 흐름 (2단계)

VAD speech_start → N ms 지속? → Soft: TTS 재생 즉시 처리 (멈춤 또는 볼륨↓) STT final 도착 → 의미 있음? ─ Yes → Hard: generation abort └─ No (잡음/짧은 backchannel) → Soft 해제, 재생 복구

결정 2-A — Soft가 무엇인가

A안 · Volume Duck

TTS 출력 −12dB로 즉시 감소. 음성 합성·재생은 계속.

+ 사용자가 "들으면서 끼어들기" 가능
− 모바일 스피커에선 −12dB 효과 약함

B안 · Pause
추천

현재 phrase의 AudioBufferSourceNode 일시정지. 복구 시 같은 chunk부터 이어 재생.

+ 모바일에서도 명확히 체감
+ AudioContext 구현 단순

📦 작업량 ~3~4일 (local-interrupt-controller.ts + playback)

결정 3

Visible Ledger / beatId

화면 변화를 어느 시그널과 묶어서 "확정 / 잠정 / 롤백"으로 다룰지. 음성쪽 audible ledger는 "playback.ended 받은 phrase만 history" 결정됨. 화면쪽이 비어있음.

지금 시그널 흐름

모델 tool.call(set_scene) → 화면에 즉시 그려짐 (영구) ↕ 같은 turn 안에서 phrase는 별도로 큐잉 phrase.audio → playback.started → playback.ended └─ interrupt 시 abort

📍 문제: tool.call로 그려진 scene이 phrase abort와 무관하게 화면에 남음. 끼어들기 후 새 scene이 그 위에 덧대지면서 화면 어수선.

결정 후 시그널 흐름

tool.call(set_scene) → scene을 provisional로 그림 (beatId 부착) └─ 같은 beatId의 playback.started 또는 playback.ended 시점에 commit └─ 같은 beatId의 phrase가 abort → rollback or supersede

차니가 골라야 할 것 — beat commit policy

A · immediate

tool.call 즉시 확정. interrupt에도 화면 그대로.

+ 단순
− 화면-음성 불일치 가능

B · on_playback_started

짝 phrase 첫 audio 재생 시점에 확정.

+ "듣기 시작 = 보기 시작"
− 재생 직전 interrupt에 약함

C · scene type별 policy
추천

기본 provisional + scene 종류별로 commit 시점 다르게.

• timeline·agenda → on_playback_started
• focus_item → immediate
• card·chart → on_playback_ended
• error·system → immediate

📦 작업량 ~5~7일 (scene state 머신 + rollback 경로)

결정 4

EchoGuard

써니 자기 TTS가 마이크로 되돌아와 VAD speech_start + STT non_final/final을 트리거하는 self-interrupt 차단. 4/13 사건 재발 방지.

지금 시그널 흐름

phrase.audio 재생 중 → 스피커 출력 ↓ (블루투스/외부 스피커) 마이크 입력으로 되돌아옴 ↓ VAD speech_start → STT non_final/final → hard abort (자기 말에 자기가 끊김)

📍 브라우저 echoCancellation: true만으로 부족. 블루투스/외부 스피커에서 새어나옴.

결정 후 시그널 흐름

VAD speech_start → EchoGuard에 confidence 질의 └─ confidence ≥ 0.7 (자기 목소리 거의 확실) → Soft/Hard 둘 다 차단 └─ confidence ≥ 0.4 → Soft만, Hard 보류 └─ confidence < 0.4 → 정상 처리

📍 false positive 안전장치: EchoGuard에 막혀도 800ms 안에 의미 있는 STT final 오면 hard abort 강제 (지연 1번 허용).

차니가 골라야 할 것 — confidence 계산 방식

A · Timing only

playback window와 speech_start 시간 겹침만 사용.

+ CPU 0
− 동시 발화에 false suppression

B · Timing + Transcript
추천

A안 + assistant text와 STT non_final/final 텍스트 유사도(trigram).

+ "내가 한 말이 그대로 돌아옴" 직접 검증
+ STT 결과 재활용 (추가 신호 0)

C · Spectrogram

TTS audio buffer ↔ mic input 음파 직접 cross-correlation.

+ 가장 정확
− WASM·CPU·구현 비용 큼

📦 작업량 ~4~5일 (B안 기준, 신규 echo-guard.ts)

한 줄로 답할 수 있는 형태

  1. 1. Endpointing — continuation marker 시 settle 연장 → A +800 / B +1200 (추천) / C +1500
  2. 2-A. Soft 동작 → A Duck / B Pause (추천)
  3. 2-B. Soft 트리거 (speech_start 후) → 150 / 200ms (추천) / 250
  4. 3. Visible ledger commit policy → A immediate / B on_playback_started / C scene type별 (추천)
  5. 4. EchoGuard confidence → A timing / B timing+transcript (추천) / C spectrogram
