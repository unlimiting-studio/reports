Before / After

v3 step 1~4.7(ai-sdk 시절) vs 신규 토폴로지. 토글로 비교.

Before — ai-sdk streamText 통과 (폐기 대상)

// queue.ts (구)
const result = streamText({
  model: claudeCode("sonnet"),       // ai-sdk LanguageModelV3 어댑터
  tools: deps.tools,
  stopWhen: stepCountIs(deps.maxSteps),
  prompt,
});
silenceStreamTextRejections(result);  // 14 필드 .catch 등록 — getter access 가 stream 강제 소비
await safePostStream(thread, result);

⚠ 4 가지 문제가 한 추상층에서 발생:

  • · .text / .steps / .finishReason getter 접근만으로 consumeStream() 호출
  • · chat-sdk fullStream 받기 전에 base stream 이 이미 소비 시작 → tee + buffered
  • · controller.abort 시 background promise 14개 unhandled rejection
  • · abortSignal listener race — 운영 안전망(process-level handler)으로 가까스로 보완

After — claude-agent-sdk / codex-sdk 직접

// queue.ts (신규)
import { query } from "@anthropic-ai/claude-agent-sdk";

const session = query({
  prompt,
  options: { abortController: deps.controller, /* tools/hooks/... */ },
});

for await (const msg of session) {
  // chat-sdk thread.post 에 chunk 직접 forward
  await postChunk(thread, msg);  // SDKMessage → text-delta 매핑
}
// abort 시 SDK 가 자체 처리. silence 우회 0건. background promise 0건.

✓ 4 가지 문제가 원천적으로 사라짐:

  • · SDK 가 stream 을 AsyncGenerator 로 직접 노출 — getter sideeffect 없음
  • · options.abortController 로 명시적 abort — race 없음
  • · agent loop (tool / multi-step / stop) 가 SDK 안에 통합 — 우리 추상 불필요
  • · session 관리 (fork/resume) 도 SDK 가 제공

왜 ai-sdk 어댑터 폐기?

코드 직독 결과. ai-sdk 의 streamText 결과 객체에서 lazy getter 접근만으로 internal stream 이 강제 소비돼요.

node_modules/ai/dist/index.mjs:7655–7657 
get steps() {
  this.consumeStream();   // ← getter access → 즉시 stream 소비 시작
  return this._steps.promise;
}

.text / .usage / .finishReason / .toolCalls 등 거의 모든 derived getter 가 결국 이 steps getter 통과.

packages/app/src/runtime/handlers/utils.ts (구) 
const fields = [
  result.text,          // ← 이 access 하나가
  result.usage,         //    consumeStream 강제 호출 트리거
  result.finishReason,
  result.toolCalls,
  ...                   // 14개 필드 전부 access
];
for (const p of fields) {
  if (!p) continue;
  Promise.resolve(p).catch(() => {});
}

우리는 "unhandled rejection 차단" 만 의도였지만, 부수효과로 base stream 이 chat-sdk 보다 먼저 소비됨.

한 추상층에서 우회 3건+ = 추상층 misfit 신호. 이번엔 4건(silenceStreamTextRejections / consumeStream / abortSignal race / background promise) — 개별 fix 로 잡고 있던 게 다 같은 데서 왔어요.
