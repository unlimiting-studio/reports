Sunny Output Contract v2
& Turn Ownership

모델은 평문으로 말하고, 화면은 도구로 그린다.
한 turn은 Gateway 안의 단일 owner가 끝까지 소유한다.

Section 1

출력 모델의 변화

DEPRECATED

v1 — prefix-JSONL narration

모델이 자체 protocol을 직접 출력

SAY 안녕하세요, 일정 정리해드릴게요.
SCENE {"kind":"timeline","items":[...]}
DELTA {"ops":[{"type":"focusItem"...}]}
SAY 다음 일정은 3시예요.
✗ 훈련 분포·도구 생태계와 정합 ❌
✗ 모델이 자체 출력 형식을 자연스럽게 받지 않음
✗ tool schema 같은 코드 레벨 검증 어려움
v2 STANDARD

표준 message + tool

assistant text는 음성, scene은 tool 호출

// assistant.text
"안녕하세요, 일정 정리해드릴게요."

// tool.call
set_scene({
  kind: "timeline",
  items: [...]
})

// tool.call
focus_item({ itemId: "next" })

// assistant.text
"다음 일정은 3시예요."
✓ system / user / assistant / tool 표준
✓ 훈련 분포 그대로, 모델이 자연스러움
✓ tool schema로 코드 검증·dedup 가능
💡

왜 폐기하는가

현대 LLM은 system / user / assistant / tool 4가지 message role로 훈련됐고, function calling이 도구 사용의 표준이에요. SAY/SCENE/DELTA는 자체 prefix-JSONL이라 모델 훈련 분포 밖이고, 지킬 동기가 약해요. v2가 폐기하는 이유는 훈련 분포·도구 생태계와의 정합을 맞추는 것이에요. 운영 중 보고된 특정 사례(예: "같은 턴 두 번 말함", "turn commit 누락")의 자동 해결을 보장하지는 않아요. 그 사례들의 원인은 Railway 로그·코드로 따로 검증해야 해요.

Section 3

Voice Loop — 세 머신 + 입력 dispatcher 토폴로지

음성 루프는 Web Client (LocalTurnDetector) 머신과 Gateway (SessionRuntime) 머신, 두 개의 분리된 상태 머신이 turn.commit 하나로 연결되는 구조예요. VAD/STT 같은 입력 신호는 Client 머신 안의 transition guard로, Gateway는 자기 이벤트만 받아요.

Section 5

6 Constraints

새 contract의 명시 규칙

OUT-001 contract

SAY/SCENE/DELTA 신규 도입 금지

모델-facing 출력 계약에 도입 안 함. 코드 표면의 DELTA prefix, SCENE delta JSON, prefix-JSONL parser도 같은 폐기 대상.

OUT-002 UX

assistant text = 기본 음성 출력

모델이 평문 텍스트로 답하면 그대로 TTS로 흘러간다. SAY 같은 prefix 없이.

OUT-003 UI mutation

scene/diff = tool 호출만

화면 구성·조작은 모두 tool call로. narration string에 숨기지 않음.

OUT-004 migration

legacy alias = migration shim만

전환용 adapter로만 존재. 새 producer / 새 prompt에서 규범 ❌.

OUT-005 ★ ownership

turn 시간 축 = Gateway 단일 owner

owner 외부(Web Client, AgentRuntime, Tool, TTS, Playback)는 권위 있는 결정 ❌. 이벤트만 올리거나 owner 결정 mirror. LocalTurnDetector 같은 사전 추론은 latency 절감용.

OUT-006 ★ naming

wire 이벤트는 tool projection

turn.scene / turn.delta는 tool result projection이지 모델 narration output 아님. 이름 충돌 명시적 분리.

Section 6

Acceptance Criteria

8개 검증 시나리오

AC-1 assistant text only turn → SAY 없이도 음성 재생됨
AC-2 scene tool call → SCENE 없이도 클라이언트가 scene 갱신
AC-3 UI mutating tool call → narration string 패치 없이 UI 변경
AC-4 text + tool 혼합 → 같은 turn에 음성과 UI 효과 동시 발생
AC-5 새 producer가 SAY/SCENE/DELTA 출력 → fail-fast 또는 migration boundary 거부
AC-6 scene payload schema 위반 → drop, 클라이언트까지 도달 ❌
AC-7 ★ turn boundary / commit 결정에서 client·gateway 충돌 → gateway owner 결정이 우선, client mirror
AC-8 ★ 새 producer/reader가 wire turn.scene/turn.delta → tool projection으로만 해석, 모델 출력 형식으로 재해석 ❌
