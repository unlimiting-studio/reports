step 4.5 롤백 fe91d92

history-aware schedule turn 시도 → 두 회귀로 롤백

차니 우려 ("state 가 chat-sdk 에 저장되는데 cron 이 그걸 우회하면 다른 채팅 간 호환성 깨지지 않을까") 를 받아 cron 발화에 thread/channel history 합류 시도. codex review round 2 에서 두 회귀 surface:

  1. 채널 dispatch: channel.messages 전체를 history 로 넣으니 채널의 임의 잡담이 모델 input 에 섞임.
  2. thread dispatch: cron prompt 가 thread 에 안 남고 응답만 남아 다음 발화 history 가 assistant-only 비대칭.

두 결함 모두 cron 의 본질 (incoming message 없는 trigger) 에서 오는 구조적 문제. 차니 우려는 outbound (cron 응답이 chat-sdk state-pg 에 일반 mention 응답과 똑같이 기록) 만으로 이미 충족 — 한 갈래로 분해해보니 inbound history-aware 는 cron 본질에 안 맞음. 롤백 + docstring 에 결정 근거 박음.

부수 발견 (운영 측면)

① state-pg 의 작업 thread subscription leak

일반 mention 핸들러가 자동 subscribe 하는 chat-sdk 디자인 + 활발한 작업 thread 가 만나면, 봇이 그 thread 의 모든 후속 메시지 를 흡수해 inFlight 폭주. 5/10 첫 부팅 시 차니/세나가 작업 thread (1778332211.565639) 에서 lily 멘션 검증한 흔적이 영구 박혀, 그 후 차니/내가 보고 메시지 박을 때마다 lily 가 onSubscribedMessage 로 받아 처리 시도 → inFlight=4까지 쌓임 → steering interrupt 다발 발생.

즉시 격리: chat_state_subscriptions 테이블에서 작업 thread subscription 수동 DELETE.

step 5+ 후보: ① Slack connector spec triggers.filter(event)(차니가 4/4 박은 spec) 정식 구현 — channelId/threadTs 기준 무시 옵션 / ② 자동 unsubscribe 도구 / ③ opt-in subscribe 모드 (mention 시 자동 subscribe 비활성).

② cron incoming 합성 가설 (차니 발화 → 검토)

차니가 step 4.5 롤백 후 던진 가설 — "cron 이라도 incoming 메시지를 우리가 임의로 포맷을 정해서 만들 수도 있는 거잖아?". 검토 결과:

  • chat-sdk ThreadHistoryCache.append(threadId, message) API 는 노출돼 있음.
  • 단 이 cache 는 native history 없는 어댑터(Telegram·WhatsApp 등)용 backfill — Slack 은 conversations.history 기반이라 thread.messages 가 native history 를 그대로 가져옴 → 가짜 inject 한 메시지가 Slack 에서는 안 보이는 유령 history 가 됨.
  • 진짜로 깨끗하려면 cron prompt 를 visible user-like 메시지로 채널/thread 에 post 해야 — UX 노이즈 (봇이 자기에게 명령하고 자기 답하는) 발생. step 5+ 영역으로 미룸.
